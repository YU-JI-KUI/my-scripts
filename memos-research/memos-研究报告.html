MemOS 技术调研与选型分析

面向 LLM 记忆管理的开源方案 MemOS 调研，及其与我们现有 ark-agentic 记忆子系统的对比

MemOS 是近期记忆（Memory）方向讨论度较高的一个开源项目，提出了把记忆当作"操作系统级资源"来统一管理的思路。我们对它的定位、架构、依赖和实测表现做了一轮调研，并结合我们自研的 ark-agentic 记忆子系统，梳理了两者在设计取向上的差异，以及哪些思路值得我们吸收、哪些实现方式并不适合我们当前的技术路线。本文汇总这次调研的结论。

1 · 结论摘要

MemOS 的抽象层次和野心都很高，代表了记忆方向上一种比较系统化的思路，其概念框架值得我们学习。但需要看清楚一点：它和我们的 ark-agentic 记忆子系统并不在同一个定位层面上——

维度MemOSark-agentic 记忆子系统
本质定位记忆的操作系统 / 基础设施平台agent 框架内的一个轻量记忆子系统
覆盖范围统一明文、激活态（KV-cache）、参数三类记忆专注明文记忆一类，做扎实
召回方式向量 + 全文混合检索（依赖 embedding）整份记忆全量注入 prompt，不做检索
基础设施完整部署默认需向量库 + 图库复用现有数据库，无额外组件
适用场景大规模跨会话记忆、有基建运维能力的团队轻量、可控、易部署的 agent 应用

我们的判断：MemOS 在记忆分层、记忆单元化、记忆生命周期管理这几个概念设计上很有参考价值，其中一部分理念我们其实已经在实践（详见第 7 节）。但它的重基础设施实现——向量库加图库的混合检索——在我们当前"纯文本、无向量基建、轻量部署"的技术路线下，直接引入的性价比不高。我们更倾向于吸收它的设计思想来演进自己的方案，而非照搬它的架构。

2 · MemOS 是什么：定位与核心抽象

2.1 定位

MemOS（Memory Operating System）由 MemTensor 公司联合上海交通大学等团队提出，是一个面向大模型的"记忆操作系统"。它的核心命题是把记忆提升为一种可管理的一等系统资源——类比操作系统对 CPU、内存、磁盘的管理，对大模型的记忆做系统化的调度与治理。

  • 开源，Apache 2.0 许可，GitHub 仓库 MemTensor/MemOS，约 10.2k star，社区活跃。
  • 有两篇 arXiv 论文（编号 2505.22101 与 2507.03724，2025 年 5–7 月），学术表述较完整。
  • 对外提供统一的记忆操作接口：add / retrieve / edit / delete，记忆以可检视、可编辑的图结构组织。

2.2 核心抽象：MemCube

MemOS 的核心抽象是 MemCube，一个标准化的记忆基本单元。它封装记忆内容本身，同时携带来源（provenance）、版本（versioning）等元数据，并支持多个 MemCube 之间的组合、迁移与融合。这个设计让记忆从"一堆散落的文本"变成"有身份、有版本、可治理的对象"，是 MemOS 系统化思路的落点。

2.3 三类记忆分层

MemOS 最有辨识度的主张，是把记忆分成三类并统一管理：

记忆类型是什么类比
明文记忆（plaintext）外部可编辑、可检索的明文知识，类似 RAG硬盘上的文档 / 知识库
激活记忆（activation）以 KV-cache 为核心的推理中间状态，介于原始数据与模型参数之间内存里的运行态
参数记忆（parameter）编码在模型固定权重中的隐式知识固化进硬件的知识

这个分层的意义在于：目前多数记忆方案（如 mem0、Zep）基本只处理明文记忆一类。MemOS 的独特之处是把激活记忆（KV-cache）和参数记忆（权重）也纳入同一套抽象——在概念上是一张更完整的"记忆版图"。我们的 ark-agentic 目前同样只做明文一类，与 mem0 处在同一层。是否有必要扩展到激活态和参数级记忆，是一个需要单独评估的战略问题（见第 7 节）。

3 · 架构、依赖与部署重量

3.1 完整部署默认依赖向量库 + 图库

这是对我们最关键的一条工程事实。MemOS 的完整自托管部署，默认依赖两个外部服务：

  • Qdrant（向量数据库）——承担向量检索
  • Neo4j（图数据库）——承担图结构记忆

官方文档明确要求先在 Docker 中启动 Neo4j 和 Qdrant，且所有部署方式都以这两个服务运行为前提。也就是说，开箱即用的完整部署，需要额外拉起并运维一个向量库和一个图库。

需要客观补充一点：严格来说，Qdrant 只在使用 Neo4j 社区版（无原生向量索引）时才成为硬性必需——MemOS 文档中后端通过抽象层是可替换的。所以准确的说法是：向量库 + 图库是 MemOS 的默认与参考后端，并非架构上不可替换的硬绑定。但"完整部署要额外运维两个重服务"这个工程现实是成立的。

3.2 检索机制：向量 + 全文混合检索

MemOS 采用向量检索加全文检索（FTS5）的混合检索。官方描述中包含 hybrid-retrieval 标签，并说明由 Qdrant 提供的向量检索用于找到"概念上相关"的记忆。虽然 embedding 模型和向量库可以经配置更换，全文检索也提供了一条独立的关键词检索路径，但没有一种完全不依赖向量的检索模式——向量/embedding 是其检索设计的核心组成，绕不开。

3.3 也有轻量的本地部署选项

客观起见需要说明：MemOS 并非只有重部署一条路。它另外提供一个基于 SQLite、100% 本地运行的插件（Node.js，同样是全文加向量的混合检索），作为轻量替代。不过即便是这个"轻量"版本，仍然带有 embedding 加向量检索加独立运行时，比我们现有方案（一张数据库表 / 一个 markdown 文件、无检索基建）仍重一个量级。

4 · 关于它的 benchmark 表现

MemOS 在 LOCOMO（长对话记忆问答，以大模型作为评判）这个 benchmark 上宣称全面领先。论文给出的总分大致如下：

方案LOCOMO 总分（据 MemOS 论文）
MemOS-103175.80
MemOS-063073.31（四类任务均列第一）
Memobase72.01
Mem064.57
Zep / LangMem / OpenAI-Memory更低

这组数据需要谨慎看待，不宜直接当作"MemOS 明显更强"的结论。原因有四点：

  1. 均为厂商自测：这些分数是 MemOS 在自家评测流程、自选的评判模型和评分口径下，统一跑出来的（包括给各家 baseline 打分）。
  2. 各家自测互相打架：Mem0 用自己的评测流程报告的分数（66.9~91.6）远高于 MemOS 给它打的 64.57；Memobase 自测 LOCOMO 报 75.8%（恰好等于 MemOS 给自己的分），而 MemOS 表中只给它 72.01。谁来打分，结果就偏向谁。
  3. 数据集本身有瑕疵：LOCOMO 的标准答案中约有 99 个存在错误或歧义，诚实的分数上限大约只有 93–94%。
  4. 简单方案也能达到相近水平：据报道，Letta 用纯文件系统加 grep（完全不用向量）也能达到约 74%，与 MemOS 处在同一档。这说明在这个 benchmark 上，重基础设施并未换来实质性的领先。

最后一点对我们其实是有利的旁证：既然纯文件方案都能达到约 74%，说明记忆问答任务上的表现更多取决于抽取与组织的质量，而非是否引入了向量库。我们"纯文本 + 全量注入"的路线，在这类任务上并不天然吃亏。

5 · 我们现有的 ark-agentic 记忆子系统

为便于对比，先概括我们自己的实现：

  • 存储：纯文本、按标题分节（heading-based）的 markdown，单用户一份整体存储，有 16KB 上限。
  • 召回：整份记忆（截断到约 2000 token 预算内）直接注入系统提示词，不做检索——工具层明确不提供 search / get 接口。
  • 抽取：每轮对话结束后台异步抽取，由大模型输出结构化的增删改操作，遵循一套过滤规则决定"记什么 / 不记什么"。
  • 整理：周期性触发（24 小时或累计 3 个会话），由大模型合并近义标题、删除矛盾内容，并有乐观合并机制防止并发覆盖。
  • 并发与多租户：乐观锁加跨节点租约保证多节点一致；按 agent 做行级数据隔离。
  • 可观测：近期新增了 append-only 的操作日志，记录每次记忆变更的来源、动作与前后内容，用于变更归因和溯源。
  • 依赖：无 embedding、无向量库、无图库，仅依赖数据库访问层和大模型调用。存储后端可插拔。

7 · 结论与建议

7.1 值得吸收的设计思路

抛开具体实现，MemOS 有几个概念设计确实值得我们借鉴，而且成本不高——其中一部分我们已经在做：

MemOS 的设计对我们的启发
记忆单元携带来源与版本元数据我们的操作日志已经是这个思想的雏形（记录来源、触发消息、变更前后）。可进一步把"来源 / 版本"提升为记忆条目的一等属性。
记忆生命周期的显式管理（增删改统一接口 + 演化）我们已有结构化的增删改操作和周期整理机制，理念吻合。可借鉴其对"记忆随时间演化、融合"的显式建模。
记忆分层（明文 / 激活态 / 参数）作为审视自身路线的框架：让我们明确"当前有意只做明文记忆"是一个主动的技术选择，而非能力缺失。
记忆单元可组合、隔离、共享启发我们思考多 agent 下的记忆差异化——不同 agent 组合不同的记忆配置。

7.2 不建议直接引入的部分

以下几点在我们当前的技术路线与团队规模下，直接落地的性价比不高：

MemOS 的实现不建议直接引入的原因
向量 + 图的混合检索引入向量库和图库是较重的基建投入。只有当我们的记忆规模确实突破了全量注入的上限时才值得考虑，而目前远未到那个量级。
激活记忆（KV-cache 记忆）技术复杂度高、与推理引擎强耦合，投入产出比存疑，更偏向研究前沿而非当前工程刚需。
参数记忆（权重记忆）涉及模型微调与编辑，超出了记忆子系统的职责范畴。

7.3 结论

MemOS 的记忆分层与记忆单元化思想代表了这个方向上比较成熟的系统化思路，值得我们学习，其中记忆溯源、生命周期管理等理念我们已经在实践中吸收。它的向量加图混合检索适合海量记忆的检索场景；而我们当前面向的是轻量偏好类记忆，全量注入已经够用，且公开 benchmark 也表明重基础设施在记忆问答上并未带来实质优势。综合来看，我们的建议是：借鉴 MemOS 的概念框架来持续演进我们的记忆方案，同时不引入它的重基础设施，从而守住 ark-agentic 轻量、易部署这一核心优势。
